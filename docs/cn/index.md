# Claude Code 原理与实践：从零构建 AI 编程助手

> **教学目标**：通过 12 个渐进式课程，从最简单的循环开始，逐步构建一个完整的 AI 编程 Agent，深入理解 Claude Code 的核心原理。
> **推荐阅读方式**：先跑最小示例，再读机制解释，最后做章节练习。
> **源码位置**：每个章节对应的可运行代码在 `agents/` 目录下（如 `agents/s01_agent_loop.py`）。

---

## 核心概念速查

在开始学习之前，先建立对关键概念的精确定义。这些概念将在后续章节中反复出现，此处作为术语速查表使用。

| 概念 | 精确定义 | 首次详解 |
|------|----------|----------|
| **Agent** | 一个以 LLM 为决策核心、通过工具与外部世界交互的自主循环系统。核心结构为：`while stop_reason == "tool_use"` 的消息循环。 | s01 |
| **LLM（大语言模型）** | 接收文本输入、生成文本输出的概率模型。Agent 的"大脑"，负责理解意图、制定计划、决策工具调用。 | 00-basics |
| **工具调用（Tool Use）** | LLM 通过标准化协议表达"我想执行某个操作"的机制。LLM 只表达意图，不直接执行——执行由外部代码完成。 | 00-basics |
| **stop_reason** | LLM 每次响应返回的停止原因。`tool_use` 表示要继续循环，`end_turn` 表示任务完成，`max_tokens` 表示输出被截断。 | s01 |
| **上下文窗口（Context Window）** | 单次 LLM 调用能处理的最大 Token 数量。超过此限制需要压缩（s06）或隔离（s04）。 | 00-basics |
| **消息列表（Messages）** | 完整的对话历史，作为 LLM 的短期记忆。每次调用都发送完整列表，因为 LLM 本身无状态。 | s01 |
| **分发器（Dispatcher）** | 工具名称到处理函数的映射字典。将"调用哪个工具"的决策与"如何执行工具"的实现解耦。 | s02 |
| **路径沙盒（Path Sandbox）** | 安全机制：将所有文件操作限制在指定工作目录内，防止路径逃逸攻击（如 `../../etc/passwd`）。 | s02 |
| **TodoManager** | 内存中的任务状态管理器，帮助 Agent 在多步骤任务中保持聚焦。状态：pending → in_progress → completed。 | s03 |
| **子 Agent（SubAgent）** | 拥有独立消息列表的临时 Agent 实例。执行完子任务后销毁，只向父 Agent 返回摘要结果，保护主上下文不被污染。 | s04 |
| **技能（Skill）** | 存储在独立文件中的知识模块（SKILL.md）。通过两层注入机制按需加载，避免占用常驻上下文。 | s05 |
| **上下文压缩（Compact）** | 三层策略（微压缩 / 自动压缩 / 手动压缩）管理对话历史，实现无限长度的会话。 | s06 |
| **任务图（Task Graph / DAG）** | 持久化到磁盘的有向无环图，表示任务间的依赖关系。完成任务后自动解除下游任务的阻塞。 | s07 |
| **后台任务（Background Task）** | 在守护线程中异步执行的子进程，不阻塞主 Agent 循环。通过通知队列传递结果。 | s08 |
| **MessageBus** | Agent 间通信的消息总线，使用 JSONL 邮箱文件实现持久化的异步消息传递。 | s09 |
| **团队协议（Protocol）** | Agent 间的结构化协商机制。基于请求-响应模式，使用 request_id 关联，包含状态机（pending → approved/rejected）。 | s10 |
| **空闲循环（Idle Loop）** | 自主 Agent 在无工作时进入的轮询状态。定期检查邮箱和任务板，发现工作自动恢复执行。 | s11 |
| **Git Worktree** | Git 原生功能，允许同一仓库的多个分支同时检出在不同目录中，实现完全隔离的并行工作环境。 | s12 |

### 第一阶段：核心循环（s01-s02）

**学习重点**：理解 Agent 最本质的工作机制

| 章节 | 主题 | 核心概念 | 难度 | 源码 |
|------|------|----------|------|------|
| [00-基础篇](./00-basics.md) | 背景知识 | LLM、工具调用、上下文窗口、流式 vs 批量 | ⭐ | — |
| [s01-Agent 循环](./s01-the-agent-loop.md) | 最小可用 Agent | 消息循环、stop_reason、消息累积 | ⭐ | `s01_agent_loop.py` |

**为什么从这个阶段开始？**

Agent 的本质就是一个「读取 → 思考 → 行动 → 反馈」的循环。理解了这个循环，后面所有的机制都是在这个基础上的扩展，而不是替代。用一句话概括：**一个 `while` 循环 + 一个工具 = 一个 Agent**。

**阶段自检**：完成 s02 后，你应该能不看文档、30 分钟内从零写出一个有 3 个工具的 Agent。

### 类比说明

- **s03** 就像给 Agent 一个记事本，把大任务拆成小步骤
- **s04** 就像项目经理把子任务分发给团队成员，自己只看结果摘要
- **s05** 就像按需加载的插件系统，不占用常驻内存
- **s06** 就像大脑的短期记忆清理，保留重要信息，压缩历史细节

**阶段自检**：完成 s06 后，你应该能解释"为什么需要子 Agent"、"为什么需要压缩"、"为什么技能要按需加载"这三个问题，并各举出 2 个真实场景。

## 核心设计原则

贯穿整个课程的设计哲学：

### 1. 循环不变

> **核心原则**：Agent 循环一旦建立，后续所有机制都不修改循环本身。

```python
# 这个循环从 s01 到 s12 完全不变
while stop_reason == "tool_use":
    response = call_llm(messages, tools)
    messages.append(response)
    results = execute_tools(response)
    messages.append(results)
```

12 个章节只是在「调用 LLM 之前」和「执行工具之后」添加逻辑，核心循环保持不变。具体来说：

| 章节 | 循环前添加 | 循环后添加 |
|------|-----------|-----------|
| s03 | — | 检查是否调用了 todo，注入提醒 |
| s04 | — | 处理 task 工具，启动子 Agent 循环 |
| s05 | — | 处理 load_skill 工具，返回技能内容 |
| s06 | 微压缩、Token 检查 | 处理 compact 工具 |
| s08 | drain 通知队列 | — |
| s11 | — | 空闲轮询逻辑 |

### 2. 渐进增强

每个章节只添加**一个**新机制，且有明确的**核心口号**：

| 章节 | 口号 | 新增机制 |
|------|------|----------|
| s01 | 一个循环 + 一个工具 = Agent | 消息循环 |
| s02 | 添加工具 = 添加处理器 | 分发器 + 路径沙盒 |
| s03 | 没有计划的 Agent 会迷失 | TodoManager + 提醒 |
| s04 | 大任务拆解，每个子任务获得干净上下文 | 子 Agent |
| s05 | 需要时加载知识，而非预先加载 | 两层注入 |
| s06 | 上下文会填满，需要腾出空间 | 三层压缩 |
| s07 | 大目标拆成小任务，排序，持久化 | DAG + 磁盘 |
| s08 | 慢操作后台运行，Agent 继续思考 | 守护线程 |
| s09 | 任务太大时，委托给队友 | MessageBus + JSONL 邮箱 |
| s10 | 队友需要共享通信规则 | 请求-响应协议 |
| s11 | 队友看板并自己认领任务 | WORK/IDLE 循环 |
| s12 | 各自工作在自己的目录，互不干扰 | Git Worktree |

### 3. 可测试性

每个章节都提供可运行的 Python 脚本，你可以：

```bash
# 运行最简单的 Agent（只有 bash 工具）
uv run python agents/s01_agent_loop.py

# 运行有工具分发器的 Agent
uv run python agents/s02_tool_use.py

# 运行完整版（所有 12 章机制集成）
uv run python agents/s_full.py
```

> **注意**：运行前需要在项目根目录的 `.env` 文件中配置 API 密钥。参考 `.env.example` 了解需要哪些环境变量。

## 与生产系统的对比

本课程是一个 **0→1 的学习项目**，专注于教学核心原理。生产系统（如 Claude Code）在以下方面更加完善：

| 方面 | 本课程 | 生产系统 |
|------|--------|----------|
| 事件/钩子系统 | 最小实现 | 完整事件总线 + 钩子回调 |
| 权限治理 | 简单沙盒 + 危险命令黑名单 | 规则引擎 + 审批流 + 用户确认 |
| 会话恢复 | 基础支持（JSONL 转录） | 完整的生命周期管理 |
| MCP 协议 | 未涉及 | 完整实现（外部工具接入） |
| 流式输出 | 未涉及 | 实时流式响应 |
| 错误恢复 | 基础异常捕获 | 多层重试 + 降级策略 |

> **建议**：先掌握本课程的核心原理（占比 80%），再根据需要学习生产系统的扩展特性。
