# Claude Code 原理与实践：从零构建 AI 编程助手

> **教学目标**：通过 12 个渐进式课程，从最简单的循环开始，逐步构建一个完整的 AI 编程 Agent，深入理解 Claude Code 的核心原理。
> **推荐阅读方式**：先跑最小示例，再读机制解释，最后做章节练习。
> **源码位置**：每个章节对应的可运行代码在 `agents/` 目录下（如 `agents/s01_agent_loop.py`）。

---

## 核心概念速查

在开始学习之前，先建立对关键概念的精确定义。这些概念将在后续章节中反复出现，此处作为术语速查表使用。

> **如何使用本表**：先通读一遍建立印象，遇到不理解的定义不必纠结——后续章节会用源码和示例深入讲解。学完某个章节后，回来对照本表，确认理解是否准确。

| 概念 | 精确定义 | 本质隐喻 | 首次详解 |
|------|----------|----------|----------|
| **Agent** | 以 LLM 为决策核心、通过工具与外部世界交互的**自主循环系统**。核心结构为 `while stop_reason == "tool_use"` 的消息循环。 | 一个有手有脚的顾问：LLM 是大脑，工具是手脚，循环是不停地"思考→行动→观察→再思考"。 | s01 |
| **LLM（大语言模型）** | 基于 Transformer 架构的**自回归文本生成模型**。接收文本输入，逐个 Token 预测后续文本。本身无状态、只能处理文本。 | 一个被困在房间里的绝世聪明顾问——可以思考，但无法触碰外部世界。 | 00-basics |
| **工具调用（Tool Use）** | LLM 与外部世界交互的**标准化协议**。LLM 不直接执行操作，而是通过 `tool_use` 消息**表达执行意图**，由外部代码负责实际执行并将 `tool_result` 返回。 | 给顾问一部电话——他不能亲手操作，但可以打电话指挥别人去做。 | 00-basics |
| **stop_reason** | 每次 LLM 响应返回的**循环控制信号**。`tool_use` = 继续循环（还有活要干），`end_turn` = 退出循环（任务完成），`max_tokens` = 输出被截断。 | 红绿灯：绿灯（tool_use）继续，红灯（end_turn）停下。 | s01 |
| **上下文窗口（Context Window）** | 单次 LLM 调用能处理的**最大 Token 数量**（Claude 为 200K tokens）。包含系统提示 + 消息历史 + 本次输出，是 Agent 的"短期记忆容量"。 | 一块固定大小的黑板——写满了就得擦掉旧的才能写新的。 | 00-basics |
| **消息列表（Messages）** | 完整的对话历史，作为 LLM 的**短期记忆**。每次 API 调用都发送完整列表，因为 LLM 本身**无状态**。 | 一个不断增长的会议纪要——每次开会都要带上全部历史记录。 | s01 |
| **分发器（Dispatcher）** | 工具名称到处理函数的**映射字典**（`TOOL_HANDLERS`）。将"调用哪个工具"的决策与"如何执行工具"的实现解耦。新增工具只需在字典中添加一项，循环代码不变。 | 酒店前台：LLM 说"我要房间服务"，前台把请求转给对应部门。 | s02 |
| **路径沙盒（Path Sandbox）** | 将所有文件操作限制在指定工作目录内的**安全机制**。通过 `Path.resolve()` + `is_relative_to()` 消除 `../` 路径逃逸和符号链接攻击。 | 给 Agent 一间带锁的工作室——他只能在里面操作，不能跑到外面去。 | s02 |
| **TodoManager** | **内存中的**任务状态管理器。三态（pending → in_progress → completed）+ 单焦点约束（同一时刻只有一个 in_progress）+ 3 轮未更新提醒机制。 | Agent 的记事本——写下待办清单，做完一项划掉一项，避免迷失方向。 | s03 |
| **子 Agent（SubAgent）** | 拥有**独立消息列表**的临时 Agent 实例。执行完子任务后销毁，只向父 Agent 返回文本摘要。主上下文不被中间细节污染，Token 节省可达 96%。 | 项目经理把调研任务交给助手——助手独立完成，只交回一页摘要报告。 | s04 |
| **技能（Skill）** | 存储在 `SKILL.md` 文件中的**知识模块**。采用两层注入：Layer 1 元数据（~20 tokens/技能）常驻系统提示，Layer 2 完整内容通过 `load_skill` 工具按需加载。 | 工具箱上的标签——平时只看标签知道有什么工具，需要时才打开取出。 | s05 |
| **上下文压缩（Compact）** | **三层策略**管理对话历史：Layer 1 微压缩（每轮替换旧工具结果为占位符）、Layer 2 自动压缩（超阈值时 LLM 生成摘要替换全部历史）、Layer 3 手动压缩（Agent 主动调用 compact 工具）。 | 大脑的遗忘机制——不重要的细节模糊化，关键决策保留清晰记忆。 | s06 |
| **任务图（Task Graph / DAG）** | **持久化到磁盘**的有向无环图（`.tasks/task_N.json`）。每个任务独立文件，`blockedBy` + `blocks` 双向引用表达依赖，完成任务时 `_clear_dependency()` 自动解除下游阻塞。 | 项目管理中的甘特图——任务有先后依赖，前置完成自动解锁后续。 | s07 |
| **后台任务（Background Task）** | 在**守护线程**中异步执行的子进程。通过 `threading.Lock` 保护的通知队列向主循环传递结果，`drain` 模式在每轮 LLM 调用前统一检查。 | 快递与收件——包裹在途中你可以做别的事，到了统一去取。 | s08 |
| **MessageBus** | Agent 间通信的**消息总线**。基于 JSONL 邮箱文件（`.team/inbox/{name}.jsonl`）实现**持久化的异步消息传递**，append-only 写入 + drain 读取。 | 公司的内部邮箱系统——每人一个信箱，发信投递，收信清空。 | s09 |
| **团队协议（Protocol）** | Agent 间的**结构化协商机制**。请求-响应模式，`request_id`（UUID 前 8 位）关联请求与响应，FSM 状态机（pending → approved/rejected）。包含关机协议和计划审批协议。 | 公司的审批流程——申请提交→领导审批→确认执行，有据可查。 | s10 |
| **空闲循环（Idle Loop）** | 自主 Agent 在无工作时进入的**轮询状态**。每 5 秒检查邮箱和任务板，发现工作自动恢复 WORK；60 秒无新工作则自动 SHUTDOWN 释放资源。 | 值班保安——定时巡逻，有事处理，无事等待，超时下班。 | s11 |
| **Git Worktree** | Git 原生功能，允许同一仓库的**多个分支同时检出在不同目录中**。通过 `task_id` 绑定任务与工作树，实现文件系统级别的完全隔离。 | 同一个项目的多间独立工作室——每人一间，互不干扰，共享同一份图纸。 | s12 |

## 课程大纲

### 概念依赖图

在阅读具体章节之前，先理解各章之间的概念依赖关系。箭头表示"本章依赖前一章的概念"：

```
00-basics ──→ s01 Agent 循环 ──→ s02 工具系统
                                      │
                                      ▼
                               s03 任务规划 ──→ s04 子 Agent ──→ s05 技能系统
                                      │                               │
                                      ▼                               ▼
                               s07 任务系统 ←── s06 上下文压缩     │
                                      │                               │
                                      ▼                               │
                               s08 后台任务                         │
                                      │                               │
                                      ▼                               ▼
                               s09 Agent 团队 ──→ s10 团队协议 ──→ s11 自主 Agent
                                                                    │
                                                                    ▼
                                                             s12 工作树隔离
```

**设计规律**：每章只添加一个新机制，且不修改已有的核心循环。理解这个"渐进增强"的设计原则，比记忆代码更重要。

### 第一阶段：核心循环（s01-s02）

**学习重点**：理解 Agent 最本质的工作机制

| 章节 | 主题 | 核心概念 | 难度 | 源码 |
|------|------|----------|------|------|
| [00-基础篇](./00-basics.md) | 背景知识 | LLM、工具调用、上下文窗口、流式 vs 批量 | ⭐ | — |
| [s01-Agent 循环](./s01-the-agent-loop.md) | 最小可用 Agent | 消息循环、stop_reason、消息累积 | ⭐ | `s01_agent_loop.py` |
| [s02-工具系统](./s02-tool-use.md) | 工具调度 | 分发器模式、路径沙盒、权限控制 | ⭐⭐ | `s02_tool_use.py` |

**为什么从这个阶段开始？**

Agent 的本质就是一个「读取 → 思考 → 行动 → 反馈」的循环。理解了这个循环，后面所有的机制都是在这个基础上的扩展，而不是替代。用一句话概括：**一个 `while` 循环 + 一个工具 = 一个 Agent**。

**阶段自检**：完成 s02 后，你应该能不看文档、30 分钟内从零写出一个有 3 个工具的 Agent。

### 类比说明

- **s03** 就像给 Agent 一个记事本，把大任务拆成小步骤
- **s04** 就像项目经理把子任务分发给团队成员，自己只看结果摘要
- **s05** 就像按需加载的插件系统，不占用常驻内存
- **s06** 就像大脑的短期记忆清理，保留重要信息，压缩历史细节

**阶段自检**：完成 s06 后，你应该能解释"为什么需要子 Agent"、"为什么需要压缩"、"为什么技能要按需加载"这三个问题，并各举出 2 个真实场景。

### 设计演进

```
s03 的 TodoManager（内存）：
  └── 问题：上下文压缩后丢失
      │
      ▼
s07 的 TaskManager（磁盘）：
  └── 改进：持久化到 .tasks/ 目录，跨会话存活
  └── 新增：DAG 依赖关系，完成后自动解除阻塞

s02 的 run_bash（同步）：
  └── 问题：长时间命令阻塞 Agent 思考
      │
      ▼
s08 的 BackgroundManager（异步）：
  └── 改进：守护线程 + 通知队列
  └── 新增：超时保护、并发执行
```

**阶段自检**：完成 s08 后，你应该能设计一个跨会话的任务管理系统，画出 DAG 图，并说明任务完成后的状态传播机制。

### 设计演进

```
s09: 创建多个队友，各自独立工作（但 Lead 手动分配任务）
```

**预期收获**：能独立实现课程中的所有机制，并能从零构建一个完整的 Agent。

## 建议学习节奏

每一章都建议按同样的 3 步完成，以便把"看懂"变成"会用"：

1. **先运行示例**（5 分钟）：先执行该章脚本，观察输入、工具调用和输出。如果环境未配置，先完成 [开始前准备](#开始前准备)。
2. **再理解机制**（15-30 分钟）：带着运行结果回看章节中的架构图和关键代码。重点理解"为什么这样设计"而不仅是"怎么写"。
3. **最后做练习**（15-30 分钟）：至少完成 1 个练习，把机制迁移到你自己的场景。

> 如果你时间有限，可以只保留第 1 步和第 2 步；如果你追求工程落地，建议 3 步都做。

## 核心设计原则

贯穿整个课程的设计哲学。理解这些原则比记住具体代码更重要——代码会变，原则不会。

### 1. 循环不变

> **核心原则**：Agent 循环一旦建立，后续所有机制都不修改循环本身。

```python
# 这个循环从 s01 到 s12 完全不变
while stop_reason == "tool_use":
    response = call_llm(messages, tools)
    messages.append(response)
    results = execute_tools(response)
    messages.append(results)
```

12 个章节只是在「调用 LLM 之前」和「执行工具之后」添加逻辑，核心循环保持不变。具体来说：

| 章节 | 循环前添加 | 循环后添加 |
|------|-----------|-----------|
| s03 | — | 检查是否调用了 todo，注入提醒 |
| s04 | — | 处理 task 工具，启动子 Agent 循环 |
| s05 | — | 处理 load_skill 工具，返回技能内容 |
| s06 | 微压缩、Token 检查 | 处理 compact 工具 |
| s08 | drain 通知队列 | — |
| s11 | — | 空闲轮询逻辑 |

### 2. 渐进增强

每个章节只添加**一个**新机制，且有明确的**核心口号**：

| 章节 | 口号 | 新增机制 |
|------|------|----------|
| s01 | 一个循环 + 一个工具 = Agent | 消息循环 |
| s02 | 添加工具 = 添加处理器 | 分发器 + 路径沙盒 |
| s03 | 没有计划的 Agent 会迷失 | TodoManager + 提醒 |
| s04 | 大任务拆解，每个子任务获得干净上下文 | 子 Agent |
| s05 | 需要时加载知识，而非预先加载 | 两层注入 |
| s06 | 上下文会填满，需要腾出空间 | 三层压缩 |
| s07 | 大目标拆成小任务，排序，持久化 | DAG + 磁盘 |
| s08 | 慢操作后台运行，Agent 继续思考 | 守护线程 |
| s09 | 任务太大时，委托给队友 | MessageBus + JSONL 邮箱 |
| s10 | 队友需要共享通信规则 | 请求-响应协议 |
| s11 | 队友看板并自己认领任务 | WORK/IDLE 循环 |
| s12 | 各自工作在自己的目录，互不干扰 | Git Worktree |

### 3. 可测试性

每个章节都提供可运行的 Python 脚本，你可以：

```bash
# 运行最简单的 Agent（只有 bash 工具）
uv run python agents/s01_agent_loop.py

# 运行有工具分发器的 Agent
uv run python agents/s02_tool_use.py

# 运行完整版（所有 12 章机制集成）
uv run python agents/s_full.py
```

> **注意**：运行前需要在项目根目录的 `.env` 文件中配置 API 密钥。参考 `.env.example` 了解需要哪些环境变量。

## 与生产系统的对比

本课程是一个 **0→1 的学习项目**，专注于教学核心原理。生产系统（如 Claude Code）在以下方面更加完善：

| 方面 | 本课程 | 生产系统 | 学习价值 |
|------|--------|----------|----------|
| 事件/钩子系统 | 最小实现 | 完整事件总线 + 钩子回调 | 理解事件驱动架构的扩展性 |
| 权限治理 | 简单沙盒 + 危险命令黑名单 | 规则引擎 + 审批流 + 用户确认 | 理解安全是分层的 |
| 会话恢复 | 基础支持（JSONL 转录） | 完整的生命周期管理 | 理解持久化对用户体验的意义 |
| MCP 协议 | 未涉及 | 完整实现（外部工具接入） | 理解工具发现的标准化 |
| 流式输出 | 未涉及 | 实时流式响应 | 理解批量与流式的权衡 |
| 错误恢复 | 基础异常捕获 | 多层重试 + 降级策略 | 理解生产级健壮性 |

### 本课程的设计取舍

| 选择 | 原因 |
|------|------|
| **单文件章节**而非模块化项目 | 降低上手门槛，每章独立可运行 |
| **批量 API**而非流式 | 循环控制更清晰，教学意图更明确 |
| **文件系统**而非数据库 | 避免外部依赖，概念映射直观（目录=表，文件=行） |
| **同步循环**而非异步 | 循环逻辑最直观，调试最简单 |
| **Python**而非 TypeScript | API 调用更简洁，不需要编译步骤 |

> **建议**：先掌握本课程的核心原理（占比 80%），再根据需要学习生产系统的扩展特性。核心原理是稳定的，扩展特性会随版本迭代变化。

## 常见卡点导航

当你学习或实操时遇到问题，可以直接按"现象"跳转。每个卡点都标注了**诊断步骤**，帮助你快速定位问题层级：

| 现象 | 可能原因 | 诊断步骤 | 推荐阅读 |
|------|----------|----------|----------|
| Agent 反复调用工具但不停止 | `stop_reason` 逻辑错误或工具定义问题 | 检查 `while` 条件是否正确；检查工具 schema 是否返回了让 LLM 困惑的结果 | [s01-Agent 循环](./s01-the-agent-loop.md) |
| 不确定工具该如何定义与分发 | 缺少分发器模式理解 | 对比 s01 的 if/elif 和 s02 的字典分发，理解解耦的价值 | [s02-工具系统](./s02-tool-use.md) |
| 多步骤任务容易中途跑偏 | 缺少规划和提醒机制 | 检查是否使用了 TodoManager；观察 `rounds_since_todo` 计数器行为 | [s03-任务规划](./s03-todo-write.md) |
| 对话太长后 Agent "变笨" | 上下文饱和，重要信息被淹没 | 用 `estimate_tokens()` 检查上下文大小；观察微压缩是否生效 | [s06-上下文压缩](./s06-context-compact.md) |
| 想让任务跨会话保持状态 | 缺少持久化，状态仅存在内存 | 检查 `.tasks/` 目录是否正确生成 `task_*.json` 文件 | [s07-任务系统](./s07-task-system.md) |
| Agent 执行慢操作时卡住 | 同步阻塞，Agent 循环在等命令完成 | 对比 s02 的 `subprocess.run` 和 s08 的 `daemon=True` 线程 | [s08-后台任务](./s08-background-tasks.md) |
| 需要 Agent 间协作和隔离 | 单 Agent 限制 | 确认是"临时委托"（用子 Agent）还是"长期协作"（用团队） | [s09-Agent 团队](./s09-agent-teams.md) 和 [s12-工作树隔离](./s12-worktree-task-isolation.md) |
| 对话上下文被工具结果污染 | 缺少上下文隔离 | 检查大文件是否直接读取到主上下文而非通过子 Agent | [s04-子 Agent](./s04-subagent.md) |
| 需要大量背景知识但不想占满上下文 | 缺少按需加载机制 | 检查 SKILL.md 是否被全量注入而非两层注入 | [s05-技能系统](./s05-skill-loading.md) |
| 多个 Agent 修改同一文件 | 缺少执行隔离 | 检查是否为每个并行任务创建了独立的 Git Worktree | [s12-工作树隔离](./s12-worktree-task-isolation.md) |
| Agent 不按预期使用工具 | 系统提示不够明确或工具描述模糊 | 检查 `SYSTEM` 提示是否有明确引导；检查工具 `description` 是否包含触发条件 | [00-基础篇](./00-basics.md) 的系统提示章节 |
| 工具调用报错导致循环中断 | 异常未被捕获 | 确认每个工具处理函数都有 `try/except` 兜底，返回字符串错误信息 | [s02-工具系统](./s02-tool-use.md) 的错误处理 |
